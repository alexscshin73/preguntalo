# manual-rag-mvp 재개발 참고 문서

이 문서는 현재 저장소에 남아 있는 `manual-rag-mvp` 소스를 기준으로, 이후 같은 기능을 다시 개발할 때 빠르게 맥락을 복원할 수 있도록 정리한 참고 문서다. 분석 기준일은 2026-03-13이다.

## 1. 한 줄 요약

현재 저장소는 `manual.docx` 기반의 한국어/스페인어 매뉴얼 검색형 RAG MVP 초안이다. 핵심 아이디어는 다음과 같다.

- Word 매뉴얼을 문단 단위로 읽는다.
- 한국어/스페인어가 번갈아 나온다고 가정해 `(ko, es)` 쌍을 만든다.
- 각 문단을 OpenAI 임베딩으로 벡터화해서 ChromaDB에 저장한다.
- 사용자 질문이 들어오면 유사 문단을 검색하고, 그 문단만 근거로 답변을 생성한다.

다만 현재 저장소는 실행 가능한 완성본이라기보다, 인덱싱 스크립트와 UI 예시 코드가 한 파일에 섞여 있는 스냅샷에 가깝다.

## 2. 현재 저장소 구성

```text
manual_rag_mvp/
├── ingest.py
├── data/
│   └── manual.docx
└── .venv/
```

### 파일별 의미

`ingest.py`

- 원래 의도는 Word 문서를 읽어 ChromaDB에 적재하는 인제스트 스크립트다.
- 하지만 파일 하단에 `app.py` 예시 코드 설명문이 그대로 붙어 있어 현재는 문법 오류가 발생한다.

`data/manual.docx`

- 한국어/스페인어 병기 매뉴얼 원본이다.
- 추출 텍스트 기준으로 로그인, POS 선택, 개점 시재 입력, 매니저 선택, 영수증 서명 정도의 초기 운영 절차가 들어 있다.

`.venv`

- 로컬 개발 흔적은 남아 있지만, 현재 코드가 요구하는 의존성이 완전히 맞는 상태는 아니다.

## 3. 문서 데이터 특성

`data/manual.docx`의 텍스트를 추출해 보면 다음 특징이 있다.

- 문서 길이는 크지 않다.
- 비어 있지 않은 줄 기준 약 31개 수준의 짧은 안내 문장으로 구성되어 있다.
- 한국어 다음 줄에 스페인어가 오는 구조가 많다.
- 그러나 모든 줄이 완전한 `ko-only` 또는 `es-only`는 아니다.
- 제목 줄은 `시작하기 (Primeros Pasos)`처럼 한 줄에 두 언어가 함께 들어간다.
- 일부 설명 줄도 한 줄 안에 한국어와 스페인어가 같이 들어간다.
- `💡` 같은 시각적 구분용 요소가 독립 문단으로 존재한다.

### 실제 포함된 주제 예시

- 로그인
- POS 선택
- 주의사항
- 개점 시재 입력
- 매니저 선택 및 개점 완료
- 영수증 서명

### 재개발 시 중요한 해석

현재 문서는 "완벽하게 번갈아 나오는 다국어 병렬 코퍼스"가 아니다. 따라서 재개발 시에는 단순히 "한 줄 읽고 다음 줄과 묶기"보다 아래 단계가 먼저 필요하다.

- 문단 정규화
- 불필요한 장식 문단 제거
- 제목/본문 분리
- 한 줄 내 혼합 언어 분리 여부 판단
- 병렬 문장 정렬 규칙 정의

## 4. 현재 코드가 의도한 동작

`ingest.py` 상단의 실제 파이프라인은 아래와 같다.

### 4.1 인제스트 흐름

1. `data/manual.docx`를 읽는다.
2. 빈 문단을 제외하고 텍스트 문단만 리스트로 모은다.
3. 한글이 있으면 `ko`, 없으면 `es`로 언어를 판정한다.
4. 인접한 두 문단의 언어가 다르면 하나의 `pair_id`로 묶는다.
5. 각 문단 텍스트를 `text-embedding-3-large`로 임베딩한다.
6. ChromaDB `manual_chunks` 컬렉션에 문서/임베딩/메타데이터를 저장한다.

### 4.2 저장 메타데이터

현재 코드가 Chroma에 넣으려는 메타데이터는 다음 정도다.

- `pair_id`
- `lang`
- `source = manual.docx`

### 4.3 질의응답 UI 설계 의도

`ingest.py` 하단에는 실제 실행 파일은 아니지만 Streamlit UI 예시가 포함되어 있다. 의도된 흐름은 아래와 같다.

1. 사용자의 질문 언어를 감지한다.
2. 질문을 임베딩한다.
3. ChromaDB에서 상위 유사 문단을 검색한다.
4. 검색된 문단을 근거로 `gpt-4.1-mini`에게 답변을 요청한다.
5. 답변과 함께 근거 문단을 화면에 보여준다.

즉 전체 MVP 개념은 `문서 인덱싱 -> 벡터 검색 -> 근거 기반 답변 생성 -> Streamlit UI 출력`이다.

## 5. 현재 상태에서 확인된 문제점

재개발 전에 반드시 알고 있어야 할 핵심 이슈들이다.

### 5.1 `ingest.py`는 현재 그대로 실행되지 않음

파일 하단에 아래 성격의 설명문이 소스 코드 안에 들어가 있다.

- `5) 챗봇 UI(Streamlit) 코드 만들기`
- `app.py 파일 생성 후 아래 붙여넣기:`

이 때문에 `python -m py_compile ingest.py` 기준 문법 오류가 발생한다.

### 5.2 인덱싱 코드와 UI 코드가 파일 단위로 분리되지 않음

현재는 아래가 한 파일에 혼재되어 있다.

- 실제 인덱싱 로직
- UI 설계 예시
- 구현 가이드 문장

재개발 시 가장 먼저 해야 할 일은 역할 분리다.

### 5.3 의존성 정의가 불완전함

현재 코드상 필요한 패키지는 최소 아래와 같다.

- `openai`
- `chromadb`
- `python-docx`
- `streamlit`

하지만 현재 `.venv` 패키지 목록에는 `python-docx` 대신 `python-docs`가 보이며, `openai`도 확인되지 않았다. 즉 환경 재현성이 없다.

### 5.4 언어 판별 로직이 매우 단순함

현재 로직:

- 한글 포함 시 `ko`
- 아니면 `es`

이 방식은 아래 경우에 취약하다.

- 제목 한 줄에 두 언어가 같이 있는 경우
- 한국어 문장 안에 영문 UI 라벨이 포함되는 경우
- `💡` 같은 비문장 문단
- 숫자/기호만 있는 경우

### 5.5 페어링 가정이 원본 데이터와 완전히 맞지 않음

현재는 "한국어 다음 줄에 스페인어" 패턴을 가정하지만 원문에는 아래가 섞여 있다.

- 한 줄 혼합 언어
- 제목 문단
- 팁 아이콘 문단
- 독립 단락

따라서 `pair_id`가 실제 의미 있는 번역 쌍을 항상 보장하지 않는다.

### 5.6 주석과 실제 구현이 일부 어긋남

UI 예시 코드에는 "같은 `pair_id`의 반대 언어도 함께 붙여서 컨텍스트 강화"라는 주석이 있다. 하지만 실제로는 검색된 결과를 순회하며 중복만 제거할 뿐, `pair_id`를 기준으로 반대 언어 문단을 다시 조회하는 로직은 없다.

즉 아이디어는 있었지만 구현은 끝나지 않은 상태로 보는 것이 맞다.

### 5.7 프로젝트 기본 문서가 없음

현재 저장소에는 아래가 없다.

- 실행 방법 문서
- 요구사항 파일 (`requirements.txt` 또는 `pyproject.toml`)
- 환경 변수 예시 (`.env.example`)
- 데이터 처리 기준 문서

## 6. 현재 MVP가 전달하는 핵심 설계 의도

문제점은 있지만, 재개발에 참고할 가치가 있는 핵심 아이디어는 분명하다.

### 6.1 다국어 병기 문서를 그대로 RAG 자산으로 활용

번역본을 따로 만드는 대신, 한국어/스페인어 병기 매뉴얼 자체를 검색 인덱스로 사용하려는 방향이다.

### 6.2 답변은 반드시 검색 근거에 묶기

시스템 프롬프트가 "근거 문단에서만 답하고, 없으면 없다고 말하라"는 방향으로 설계되어 있다. 운영 매뉴얼 QnA에는 적절한 접근이다.

### 6.3 문서 규모가 작기 때문에 MVP는 단순 구조로 시작 가능

현재 문서 분량이 작아서, 재개발 초기에는 아래 구조만으로도 충분하다.

- 단일 문서 로더
- 규칙 기반 정규화
- ChromaDB
- OpenAI 임베딩
- 간단한 웹 UI

## 7. 재개발 시 권장 구조

아래처럼 최소 단위로 분리하는 것을 권장한다.

```text
manual_rag_mvp/
├── README.md
├── requirements.txt
├── .env.example
├── data/
│   └── manual.docx
├── app.py
├── ingest.py
├── src/
│   ├── config.py
│   ├── loaders.py
│   ├── normalizers.py
│   ├── chunkers.py
│   ├── retriever.py
│   ├── qa.py
│   └── types.py
└── chroma_db/
```

### 역할 분리 기준

`ingest.py`

- 문서 로드
- 정규화
- 청킹
- 메타데이터 생성
- 임베딩 및 Chroma 적재

`app.py`

- 사용자 입력
- 질의 임베딩
- 검색
- LLM 답변 생성
- 근거 출력

`src/normalizers.py`

- `💡` 제거/보존 규칙
- 제목 감지
- 혼합 언어 문장 분리 규칙
- 공백/기호 정리

`src/chunkers.py`

- 병렬 문장 정렬
- section 단위 chunk
- parent/child chunk 설계

## 8. 재개발 시 권장 데이터 모델

문서 한 줄만 저장하는 구조보다, 아래 메타데이터를 권장한다.

```json
{
  "doc_id": "manual_v1",
  "section_id": "opening_login",
  "pair_id": "uuid",
  "lang": "ko",
  "kind": "title | body | tip",
  "source": "manual.docx",
  "order": 12,
  "text": "처음 시스템에 접속할 때는 반드시 본인의 아이디로 로그인해야 합니다."
}
```

### 추가로 있으면 좋은 필드

- `peer_text`
- `peer_lang`
- `page`
- `heading_path`
- `ui_label`
- `version`

이렇게 해두면 검색 결과를 묶어서 보여주기 쉽고, "같은 의미의 반대 언어 문장"을 함께 노출하기도 쉬워진다.

## 9. 재개발 우선순위

### 1단계: 실행 가능한 최소 버전 복원

- `ingest.py`에서 설명문과 UI 예시를 제거한다.
- `app.py`를 별도 파일로 만든다.
- 의존성 파일을 만든다.
- `.env.example`에 `OPENAI_API_KEY`를 명시한다.

### 2단계: 문서 전처리 안정화

- 혼합 언어 줄 처리 규칙을 만든다.
- `💡` 문단 처리 기준을 정한다.
- 제목/본문/팁 유형을 나눈다.
- 단순 인접 페어링 대신 섹션 기반 정렬을 도입한다.

### 3단계: 검색 품질 개선

- `pair_id` 기준 반대 언어 동시 노출 구현
- section 단위 컨텍스트 묶기
- 중복 제거 규칙 강화
- 검색 결과 재정렬

### 4단계: 운영 가능 형태로 정리

- 로깅
- 에러 처리
- 재인덱싱 명령
- 간단한 평가 질문셋

## 10. 권장 기술 스택

현재 소스의 의도를 최대한 유지한다면 아래 조합이 가장 자연스럽다.

- Python
- OpenAI Embeddings
- ChromaDB
- Streamlit
- `python-docx`

문서가 늘어날 가능성이 있다면 이후에만 아래를 검토하면 된다.

- LangGraph 또는 자체 오케스트레이션
- PostgreSQL + pgvector
- FastAPI 백엔드 분리
- 관리자용 인덱싱 파이프라인

## 11. 재개발 시 체크리스트

- 현재 소스는 참고용 스냅샷으로 간주할 것
- `ingest.py`의 상단 로직만 핵심 자산으로 볼 것
- 하단 UI 코드는 "의도 설명"으로만 참고할 것
- 원본 문서가 완전한 병렬 번역 포맷이 아님을 전제로 설계할 것
- 실행 환경을 코드보다 먼저 정리할 것
- 검색 근거를 함께 보여주는 UX는 반드시 유지할 것

## 12. 결론

이 저장소의 본질은 "병기된 운영 매뉴얼을 빠르게 RAG QnA로 바꿔보려는 초기 MVP"다. 가장 중요한 자산은 거창한 프레임워크가 아니라 아래 세 가지다.

- `manual.docx`라는 실제 운영 문서
- 문서 쌍을 묶어 다국어 검색을 하려는 아이디어
- 근거 기반 응답만 하도록 제한한 QnA 설계

재개발은 현재 코드를 그대로 살리는 방식보다, 이 의도를 기준으로 구조를 다시 나누고 데이터 전처리를 먼저 안정화하는 방식이 가장 효율적이다.
